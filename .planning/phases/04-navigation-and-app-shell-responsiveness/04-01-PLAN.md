<objective>
As a My Recipes user, I want drawer, tab, global-search, and detail-route navigation from large lists to acknowledge my tap immediately so that daily cooking and shopping flows feel responsive without losing existing tools or information density.

Purpose: Phase 4 starts by turning the existing duplicated local route-feedback hook into one app-shell behavior and making the sidebar drawer render a cheap navigation shell before its heavier tools. This plan is the user-visible behavior slice; test/evidence expansion is handled in Plan 04-02.
Output: Shared app-shell navigation feedback, fast drawer shell, deferred drawer tool body, and app-shell navigation integration for sidebar, bottom tabs, global search, dish detail, and shopping-list detail paths.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User click to app-shell navigation | Drawer, tab, search, and list-detail clicks now share route feedback state. |
| Deferred drawer shell to sensitive admin state | Drawer body may stay mounted or deferred while admin/PIN controls remain local browser state. |
| Local route feedback to network/sync systems | Feedback is local UI state and must not expose or depend on GitHub/Gist tokens. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-D | Denial of Service | Route feedback overlay | mitigate | Suppress duplicate pending destinations and clear after path-plus-paint with fallback cleanup. |
| T-04-01-I | Information Disclosure | Drawer admin/PIN controls | mitigate | Reset PIN text/error on close if drawer persistence changes; never read or expose `.env` values. |
| T-04-01-S | Spoofing | Route feedback copy | mitigate | Keep feedback local and generic; do not imply remote sync or successful navigation before the route actually changes. |
</threat_model>
