<objective>
As a My Recipes user, I want online startup sync checks to stay behind the first usable app/list interaction so that online mode feels as responsive as offline mode while preserving local-first data and GitHub shared-data detection.

Purpose: Phase 3 starts by removing the online-only startup fetch from the urgent render path. Offline behavior already exits early; online behavior must now wait for a short app-ready and interaction-quiet window before fetching the shared manifest.
Output: Deferred shared-manifest scheduling, deterministic due/fresh/offline sync test setup, and performance evidence that list hot paths remain inside Phase 2 practical budgets while a GitHub check is pending.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser startup to GitHub Raw | Online startup may contact GitHub directly from the browser. |
| Test fixture to production hook | Seed and network fixtures control sync state without changing production data shape. |
| Service-worker cache to strict evidence | Normal checks intentionally unregister service workers for deterministic performance evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-I | Information Disclosure | Sync/test fixtures | mitigate | Use deterministic manifest/data fixtures and never read or print `.env` secret values. |
| T-03-01-D | Denial of Service | Startup manifest fetch | mitigate | Defer fetch until app-ready and interaction-quiet windows; cancel on unmount. |
| T-03-01-R | Repudiation | Online/offline startup evidence | mitigate | Write `phase3-startup-*` timing ids with network/image mode metadata and resources. |
